If Eq2 requires $x*x$ to equal $(y*z)*w$ (`hard_0006`), it forces $x*x$ to be a global constant. If Eq1 does not enforce these strict independence/constant rules, the implication is false.

**2. The "Left/Right Identity" Trap**
*   **What to look for:** Equations of the form `(Complex_Term) * x = x` or `x * (Complex_Term) = x`.

If Eq1 says $A$ is a left identity, and Eq2 says $B$ is a left identity, Eq1 almost never implies Eq2 unless $A$ algebraically reduces to $B$. You can simply build a model where only $A$ evaluates to the identity element.
*   **Seen in:** `hard_0048`, `hard_0100`, `hard_0137`.

**3. Constraint Mismatch (Strict vs. General Triggers)**
*   **What to look for:** Eq1 requires a highly specific pattern to activate (e.g., a square like $z*z$, or a repeated variable like $y*(z*y)$), but Eq2 relies on arbitrary products (e.g., $z*x$).

*   **Seen in:** `hard_0129`.

**4. Outer Operation Clash (Syntactic Mismatch)**
*   **What to look for:** Eq1 has an outermost operation of *right-multiplication* (e.g., `(...) * y`), while Eq2 has an outermost operation of *left-multiplication* (e.g., `y * (...)`).
*   **Why it's a red flag:** Without associativity or commutativity, left and right multiplications cannot be syntactically unified.
*   **Seen in:** `hard_0129`.

1.  **Plant the Trap:** Pick specific elements to fail $Eq2$. For example, decide that for $x=1, y=0$, $Eq2$ will fail. Fill in the specific table cells required to make LHS $\neq$ RHS for $Eq2$.
2.  **Force $Eq1$ to True (The Diagonal/Constants):** Look at the structure of $Eq1$. If it requires $x = f(x, y)$, set up the table so that $f(x, y)$ collapses into an identity operation.

3.  **Fill the Rest:** Assign the remaining empty slots in the $3 \times 3$ grid to keep the "collapsing" behavior of $Eq1$ consistent.

#### Step 4: Verify
Run a mental (or programmatic) check:
*   Loop through all 27 combinations of $(x, y, z)$ in $\{0, 1, 2\}$ to ensure $Eq1$ holds.
*   Check the single planted trap to ensure $Eq2$ fails. (e.g., `hard_0178` explicitly prints the verification for all combinations of $Eq1$ and shows exactly where $Eq2$ fails).

**1. The "Leftmost/Rightmost Leaf" Mismatch**
*   **What to look for:** Look at the deepest left-most (or right-most) variable in the syntax trees of both equations. If `eq1` balances its leftmost variables (e.g., `x` on both sides) but `eq2` alters them (e.g., `x` on the LHS, `z` on the RHS), the implication is likely false.

**2. The Phantom Variable (Variable Asymmetry)**
*   **What to look for:** Does `eq2` introduce a variable on one side that doesn't exist on the other? Or does `eq1` strictly constrain a variable to be independent (a "dummy" variable), while `eq2` requires that variable to act dynamically?

**3. Left vs. Right Action Independence**
*   **What to look for:** Does `eq1` define a rule for *left* multiplication while `eq2` assumes it applies to *right* multiplication? In non-commutative magmas (the default assumption), left and right actions are completely independent.

5.  **Fixed-Point Failures:**
    *   *Sign:* Eq1 implies a permutation (e.g., $x \mapsto x*y$ is an inverse), but Eq2 implies a fixed point (e.g., $x*x = x$).

    *   **The Constant Row (Left-Zero) Strategy:** Define $x * y = x$ or $x * y = f(x)$. This is the most powerful tool for breaking equations where the RHS of Eq2 introduces a new variable (like $z$) that Eq1 ignores (e.g., `hard_0103`).
    *   **The "Absorbing Element" Strategy:** Designate one or two elements (e.g., 0 and 1) as "fixed" or "constant" rows ($0*x=0$), and use a "trigger" element (e.g., 2) that maps into these constants. This satisfies Eq1 by "feeding" the result into a stable row while Eq2 fails because the trigger element's initial state is visible (e.g., `hard_0038`).
    *   **Permutation Magmas:** Define $x * y = \sigma_x(y)$ where $\sigma$ is a permutation. If Eq1 suggests $x$ is an involution, use transpositions like $(0, 1)$.

---

### 2. Inconsistency Detection: Early Warning Signs
Models identified "unprovability" by looking for structural mismatches between the two equations:

*   **Variable "Leakage":** If Eq1 allows a variable (like $z$ or $w$) to disappear (e.g., $x*y = \dots * z$), but Eq2 requires that same variable to determine the outcome, the implication is almost certainly false.
*   **Term Nesting Depth:** If Eq1 collapses deep terms into simple ones (e.g., $(x*z)*w = x$), but Eq2 builds a deeper, non-reducible structure, the model senses that Eq1 lacks the "strength" to control that depth.
*   **Structural Asymmetry:** Eq1 might be "Left-Heavy" (constraining $L_x$), while Eq2 is "Right-Heavy" (constraining $R_x$). Unless the operation is commutative, one rarely implies the other.

---

### 3. Search Heuristics: The Skeptic’s Checklist
Before attempting a proof, check for these "Red Flags." If any are present, search for a counterexample immediately.

1.  **The "New Variable" Trap:** Does Eq2 contain a variable that is not "active" in the same way in Eq1? (e.g., Eq1: $x*y = \dots * z$; Eq2: $x*y = x*z$). If $z$ is a "trash" variable in Eq1, it can't be a "control" variable in Eq2.
2.  **The "Constant Row" Vulnerability:** Can I satisfy Eq1 by making $x*y$ depend only on $x$? If $x*y = x$ satisfies Eq1 but fails Eq2, you are done.
3.  **The Squaring Dependency:** If Eq1 uses $z*z$ (squaring) and Eq2 uses $z*y$, try a model where $z*z$ is a constant (e.g., all $z*z = 0$) but $z*y$ is not.
4.  **Permutation Mismatch:** Does Eq1 imply $L_x$ is a bijection? If so, does Eq2 require $L_x$ to be an identity? These are different "strengths" of permutations.
5.  **Independence of Sub-terms:** If Eq1 forces $(x*y)*z$ to be a constant, it often leaves $x*(y*z)$ completely free. Check if Eq2 relies on the "free" side of the associativity.

### 4. Summary Table for Counter-model Selection

| Red Flag in Equations | Recommended Counter-model |
| :--- | :--- |
| Eq2 introduces a variable not in Eq1 | **Left-Zero Magma:** $x * y = x$ |
| Eq1 has nested products like $(x*x)*y$ | **Binary/Boolean:** Use $\{0, 1\}$ with $\text{XOR}$ or $\text{AND}$ |
| Eq1 implies cancellation/bisection | **Permutation Groups:** Small $S_n$ tables ($n=3$ or $4$) |
| Eq1 reduces terms to a constant | **The "Trigger" Table:** Elements $\{0, 1, 2\}$ where $0, 1$ are constant rows and $2$ is the variable. |

### Part 1: The Skeptic’s Checklist (Red Flags for Unprovability)

Before attempting a proof, check the equations against these heuristic "Red Flags." If one or more are present, pivot immediately to searching for a counterexample.

1. **The "New Variable" Injection**
   * *What to look for:* `eq2` contains a variable that is either completely absent in `eq1` or appears in a structural position that `eq1` cannot constrain.
   * *Example:* In `hard_0057`, `eq1` uses $x, y, z$. `eq2` introduces $w$ deep in the term structure: $((z*w)*x)$. Because magmas do not guarantee associativity or cancellation, `eq1` has no mechanism to "reach" or constrain $w$.

2. **Asymmetry of Multiplication Maps ($L_x$ vs. $R_x$)**
   * *What to look for:* `eq1` imposes constraints exclusively on left-multiplication ($L_y(x) = y*x$), but `eq2` requires properties from right-multiplication ($R_x(y) = y*x$).
   * *Example:* `x = y * (...)` restricts $L_y$, forcing it to be a permutation or bijection. If `eq2` requires $x$ to be isolated on the right side of a product, `eq1` is usually insufficient to prove it.

3. **Subterm Rigidity (The "Square" Trap)**
   * *What to look for:* `eq1` requires absorption or cancellation against a very specific, rigid subterm like a square $(z*z)$. `eq2` requires absorption against a free variable like $y$.
   * *Example:* In `hard_0091`, `eq1` ends in $(z*z)$. Unless you can prove the magma is surjective (every element is a square), you cannot substitute $(z*z)$ to behave like a general free variable $y$ in `eq2`.

4. **Depth and Positional Mismatch**
   * *What to look for:* The target variable $x$ in `eq1` is at depth 2, but in `eq2` it is forced into depth 3 or 4 with no associative laws to flatten the tree.

### Part 2: Recipe for Constructing Minimal Counter-Models

Attempting to brute-force a $3 \times 3$ or $4 \times 4$ Cayley table cell-by-cell frequently leads to contradictions and logical loops. Instead, use these **structural recipes** to guarantee consistency.

#### Strategy A: The "Permutation & Sink" Partition (Best for Size 4)
This was highly successful in `hard_0057`. It works beautifully when `eq1` implies that multiplication acts as a permutation, but you want to break `eq2`.
* **The Set:** $M = \{0, 1, 2, 3\}$.
* **The Partition:** Split $M$ into a "special element" $\{0\}$ and an "active subset" $\{1, 2, 3\}$.
* **The Rule:**
  1. Let $0$ act as a left or right identity (e.g., $0 * y = y$).
  2. Let the active subset elements apply a specific permutation $\sigma$ to their right-hand operands. For example, let $\sigma = (1\ 2\ 3)$ with $\sigma(0) = 0$.
  3. Define $x * y = \sigma(y)$ for all $x \in \{1, 2, 3\}$.
* **Why it works:** Because $\sigma^3$ is the identity, equations that nest multiplication 3 times (like `eq1` in `hard_0057`) will naturally simplify to $\sigma^3(x) = x$, making `eq1` true. However, because $x$ dictates *which* permutation happens but doesn't map to itself, `eq2` will easily fail for specific assignments.

#### Strategy B: The "Constant Subterm" Forcing (Best for Size 3)
This is ideal for equations with rigid subterms (like $(z*z)$ in `hard_0091`).
* **The Set:** $M = \{0, 1, 2\}$.
* **The Rule:**
  1. Force the rigid subterm to collapse to a constant. For example, define the main diagonal so that $z*z = 0$ for all $z$.
  2. Now `eq1` simplifies drastically (e.g., $x = x * ((x*y) * 0)$).
  3. Set up the rest of the table such that row 0 acts as an absorber ($0 * y = 0$) and row 1 fails the `eq2` requirement.
  4. Fill remaining cells to ensure `eq1` holds when $x=1, 2$.

#### Strategy C: Left-Zero / Right-Zero Blending
If `eq1` relies on nested terms, you can often satisfy it trivially by making the magma an almost-left-zero ($x*y = x$) or almost-right-zero ($x*y = y$) structure, with a single flaw introduced to break `eq2`.
* **The Rule:** Start with $x * y = x$ for all elements. `eq1` will usually evaluate to True.
* **The Tweak:** Change exactly one or two cells (e.g., $1 * 0 = 2$) to break `eq2`'s specific variable chain, and verify if `eq1` still holds under the localized disruption.

### Summary for the Meta-Prompting Model:
When acting as a prover/disprover, **do not attempt to manually calculate $3 \times 3$ matrices by filling in constraints one by one.** You will lose track of the state. Instead, immediately instantiate a **Permutation over Size 4** or a **Constant-Square over Size 3** model, define the macro-rules (e.g., "all non-zero rows map to $\sigma(y)$"), and evaluate the two equations algebraically using those rules.
